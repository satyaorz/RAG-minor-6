PROJECT TREEQA: IMPLEMENTATION HANDOFF
Last updated: 2026-03-08

Purpose
This document is a detailed handoff for the current TreeQA implementation so another engineer or agent can continue without rediscovering the existing architecture.

Current state
- The repository has moved beyond a scaffold.
- It now supports OpenRouter-backed LLM execution.
- It supports local file-backed ingestion and retrieval.
- It runs a tree-shaped multi-step QA pipeline.
- It has diagnostics, tests, and a basic Streamlit UI.

Core runtime flow
1. Load .env and environment variables.
2. Build providers from settings.
3. Ingest local files from `data/` into JSONL indices.
4. Decompose the query into a root node plus child nodes.
5. Retrieve evidence for each leaf node from local vector and graph indices.
6. Generate grounded sub-answers.
7. Validate sub-answers.
8. Aggregate verified child answers into a final answer.

Implemented capabilities
- `.env` auto-loading
- OpenRouter integration
- LLM fallback model chain
- Local vector and local graph backends
- `treeqa ingest`
- `treeqa doctor`
- `treeqa run`
- Tree-based pipeline state
- Streamlit logic-tree rendering
- Retrieval scoring and snippet extraction
- Local test suite covering config, adapters, diagnostics, ingest, retrieval, scoring, generator, and pipeline behavior

Most important configuration
- `TREEQA_LLM_PROVIDER=openrouter`
- `TREEQA_LLM_MODEL=arcee-ai/trinity-mini:free`
- `TREEQA_LLM_FALLBACK_MODELS=nvidia/nemotron-3-nano-30b-a3b:free,arcee-ai/trinity-large-preview:free,qwen/qwen3-4b:free,openrouter/free`
- `TREEQA_VECTOR_PROVIDER=local`
- `TREEQA_GRAPH_PROVIDER=local`
- `TREEQA_DATA_DIR=data`

Important note
- Relative paths now resolve against the repository root, not the current shell directory.
- This fixed a bug where running from `src/` caused `data/` to resolve incorrectly.

Main files and roles

Root files
- `project_spec.md`: project scope and intended roadmap.
- `README.md`: setup, commands, provider guidance.
- `pyproject.toml`: packaging and dependencies.
- `.env.example`: recommended local environment template.

Data
- `data/documents/treeqa_overview.md`: sample unstructured source text.
- `data/graph/facts.jsonl`: sample structured facts.
- `data/index/vector_index.jsonl`: generated local vector-style index.
- `data/index/graph_facts.jsonl`: generated local graph-style index.

Package entry and config
- `src/treeqa/__init__.py`: package export for `TreeQAPipeline`.
- `src/treeqa/config.py`: environment loading, path resolution, runtime settings.
- `src/treeqa/models.py`: dataclasses for documents, validations, nodes, and pipeline results.
- `src/treeqa/state.py`: pipeline execution state.

Pipeline and workflow
- `src/treeqa/pipeline.py`: main orchestration logic.
- `src/treeqa/workflow.py`: placeholder workflow factory for a future graph-based orchestration layer.
- `src/treeqa/cli.py`: command-line entrypoint.
- `src/treeqa/diagnostics.py`: provider and config health checks.
- `src/treeqa/ingest.py`: builds local JSONL indices from `data/`.

Agents
- `src/treeqa/agents/decomposer.py`: root/child query decomposition.
- `src/treeqa/agents/generator.py`: answer synthesis and final aggregation.
- `src/treeqa/agents/validator.py`: support checking via heuristics or LLM judge.
- `src/treeqa/agents/corrector.py`: retrieval-oriented query refinement.

Backends
- `src/treeqa/backends/llm.py`: OpenAI-compatible client, OpenRouter support, fallback model wrapper.
- `src/treeqa/backends/vector.py`: memory, local, and Qdrant vector backends.
- `src/treeqa/backends/graph.py`: memory, local, and Neo4j graph backends.

Retrieval
- `src/treeqa/retrieval/hybrid.py`: combines vector and graph retrieval.
- `src/treeqa/retrieval/scoring.py`: normalization, tokenization, lexical scoring, snippet extraction, document reranking.
- `src/treeqa/retrieval/__init__.py`: lazy export to avoid import cycles.

UI
- `src/treeqa/ui/streamlit_app.py`: basic Streamlit UI that runs the pipeline and renders the logic tree.

Tests
- `tests/test_adapters.py`: adapter construction and fallback behavior.
- `tests/test_config.py`: `.env` loading and repo-root path resolution.
- `tests/test_diagnostics.py`: doctor behavior and local-index failure cases.
- `tests/test_generator.py`: source formatting and text cleanup.
- `tests/test_ingest.py`: local ingestion and local backends.
- `tests/test_pipeline.py`: tree pipeline behavior.
- `tests/test_retriever.py`: hybrid retriever behavior.
- `tests/test_scoring.py`: normalization, snippets, reranking.

Current strengths
- End-to-end local workflow works.
- OpenRouter path is live.
- Local retrieval is usable without external DB setup.
- Tree execution and validation are functional.
- The project is test-backed.

Current weaknesses
- Retrieval is lexical, not semantic.
- Corpus is tiny.
- Final answers are still somewhat extractive.
- No formal benchmark/evaluation harness yet.
- No real LangGraph orchestration yet.
- `src/data/index/` contains stale generated artifacts from an earlier bug and should be cleaned up.

Recommended next steps
1. Remove stale `src/data/index` artifacts.
2. Add real documents to `data/documents`.
3. Add richer structured facts to `data/graph`.
4. Improve ingest metadata and chunk overlap.
5. Improve answer synthesis quality without losing grounding.
6. Add a benchmark/evaluation loop.
7. Replace recursive orchestration with explicit graph orchestration.

Commands that matter
- `python -m treeqa.cli ingest`
- `python -m treeqa.cli doctor`
- `python -m treeqa.cli doctor --live-llm`
- `python -m treeqa.cli run`
- `python -m unittest discover -s tests`
