======================================================================
1. PROJECT GOAL
======================================================================

This repository is building a hallucination-aware RAG system for multi-hop QA.

The intended system behavior is:
- decompose a user query into smaller verifiable sub-questions
- retrieve evidence from both unstructured and structured sources
- answer each sub-question using retrieved evidence only
- validate whether the sub-answer is actually supported
- retry or refine when support is weak
- aggregate the verified sub-answers into a final answer
- expose the logic tree in a UI so the user can inspect how the answer was formed

Current implementation status:
- The project is no longer just a bare scaffold.
- It now has a working CLI, local ingestion pipeline, local hybrid retrieval path, OpenRouter LLM support, diagnostics, a tree-shaped pipeline state, and a basic Streamlit UI.
- It is still early-stage and not yet production-grade in retrieval quality, orchestration sophistication, or evaluation rigor.

======================================================================
2. CURRENT HIGH-LEVEL ARCHITECTURE
======================================================================

The current architecture has these major layers:

1. Configuration and environment loading
- Reads environment variables from a repo-local .env automatically.
- Supports OpenAI-compatible LLM providers and several retrieval provider modes.

2. Backends
- LLM backend:
  - supports OpenAI-compatible chat APIs
  - supports OpenRouter explicitly
  - supports model fallback lists
- Vector backend:
  - memory mode
  - local filesystem index mode
  - Qdrant mode
- Graph backend:
  - memory mode
  - local filesystem index mode
  - Neo4j mode

3. Agents
- Query decomposition
- Answer generation
- Validation
- Query correction/refinement

4. Retrieval
- Hybrid retriever combining vector and graph results
- Local lexical ranking and snippet extraction helpers

5. Pipeline
- Tree-based execution
- Leaf nodes do retrieval/generation/validation
- Parent nodes aggregate child answers and child confidence

6. Diagnostics and CLI
- ingest
- doctor
- run

7. UI
- Streamlit app rendering final answer and nested logic tree cards
- FastAPI + SPA web UI served at GET / with interactive logic tree

======================================================================
3. CURRENT RUNTIME FLOW
======================================================================

Typical local flow:

1. Create/update .env
2. Put files under:
- data/documents/
- data/graph/

3. Build local indices:
- python -m treeqa.cli ingest

4. Check runtime setup:
- python -m treeqa.cli doctor
- python -m treeqa.cli doctor --live-llm

5. Run the pipeline:
- python -m treeqa.cli run

6. Or open the Streamlit UI:
- streamlit run src/treeqa/ui/streamlit_app.py

7. Or start the FastAPI web server (serves an SPA at http://localhost:8000):
- pip install -e ".[api]"
- PYTHONPATH=src uvicorn treeqa.api.app:app --reload --port 8000

Detailed execution path of "run":

1. TreeQAPipeline is constructed.
2. Settings are loaded from env and .env.
3. The pipeline builds:
- LLM client
- vector backend
- graph backend
- decomposer
- validator
- corrector
- generator

4. The query is decomposed into a root node with child leaf nodes.
5. Each leaf node:
- retrieves documents
- generates an answer
- validates support
- retries if needed

6. Parent/root nodes:
- aggregate child answers
- aggregate child confidence
- derive verified/needs_review status

7. Final output is serialized as JSON with:
- query
- root tree
- flattened leaf nodes
- final answer

======================================================================

LLM:

- The current recommended primary free model is:
  - arcee-ai/trinity-mini:free

  - nvidia/nemotron-3-nano-30b-a3b:free
  - arcee-ai/trinity-large-preview:free
  - qwen/qwen3-4b:free
  - openrouter/free

Retrieval:
- For real development right now, the preferred mode is:
  - TREEQA_VECTOR_PROVIDER=local
  - TREEQA_GRAPH_PROVIDER=local

Reason:
- It uses real project files under data/
- It avoids external DB setup while still giving a realistic retrieval workflow
- It is much more useful than pure memory stubs for prompt and logic iteration

External retrieval providers exist in code, but are not the best immediate next step:
- qdrant
- neo4j

6. IMPORTANT IMPLEMENTED BEHAVIOR
======================================================================

6.1 Tree structure
- The system no longer returns only a flat list of sub-questions.
- The decomposer returns a root QueryNode.
- Multi-part queries become root + children.
- Single-part queries use the root as the leaf.

6.2 Local ingestion
- There is now a local ingestion step.
- It reads source documents from data/documents.
- It reads structured facts from data/graph.
- It writes JSONL indices under data/index.

6.3 Local hybrid retrieval
- The pipeline can now retrieve from local indexed files.
- It combines vector-like and graph-like local results.
- Results are ranked with lexical scoring and deduped.

6.4 LLM diagnostics
- The CLI can now validate config and provider wiring.
- doctor checks provider setup.
- doctor --live-llm verifies an actual LLM round-trip.

6.5 Model fallback
- The LLM client supports a fallback model list.
- If one configured OpenRouter free model fails, the next one can be tried.

6.6 Output quality improvements
- Retrieval now uses lexical ranking, normalized text, and snippet extraction.
- Ingestion now creates smaller chunks than before.
- Final output is more concise and source-labeled than earlier versions.

======================================================================
7. CURRENT LIMITATIONS
======================================================================

This project is working, but several areas remain weak or unfinished.

7.1 Retrieval is still shallow
- The local retrieval path is lexical, not embedding-based.
- It does not use semantic search or learned reranking.
- It is only a practical bridge before external vector infrastructure.

7.2 LLM answers are still somewhat extractive
- Responses are better than before, but still often mirror source wording.
- Final synthesis is concise, but not yet strong enough for polished user-facing output.

7.3 Validation remains basic
- The validator can use an LLM judge or lexical heuristics.
- It is not yet a rigorous factuality pipeline.
- There is no NLI model, calibrated scoring, or strong judge schema enforcement.

7.4 No formal evaluation harness yet
- No dataset runner for HotpotQA / 2WikiMultiHopQA
- No RAGAS or TruLens integration
- No regression dataset

7.5 Workflow is tree-recursive, not a real LangGraph graph yet
- The current orchestration lives in Python control flow.
- The spec calls for agentic or graph-based orchestration.
- There is a placeholder workflow factory but no actual LangGraph integration yet.

7.6 The UI is still basic
- It renders nested cards and raw JSON.
- It does not yet provide a stronger interactive reasoning explorer or graph visualization library integration.

11. MOST IMPORTANT CURRENT KNOWN ISSUES
======================================================================

1. Retrieval is still lexical, not semantic
- Scoring uses BM25-style token overlap, not embedding similarity.
- Works well for domain-specific queries; fails on paraphrase or vocabulary mismatch.

2. Each API request reconstructs the full pipeline
- src/treeqa/api/app.py calls TreeQAPipeline() per request.
- This reloads local index files from disk on every request.
- Quick fix: move pipeline construction to module startup with a module-level singleton.

3. The corpus is 5 documents from one domain
- Multi-hop reasoning quality is still limited by evidence breadth.
- HotpotQA / 2WikiMultiHopQA integration would give meaningful benchmark numbers.

4. Validation still purely lexical when LLM is stubbed
- Heuristic validator relies on token overlap, which is noisy.
- NLI model (e.g., cross-encoder) would give a stronger factuality signal.

5. RAGAS / TruLens integration not yet done
- Local benchmark runner measures EM and F1 against local gold labels.
- RAGAS faithfulness / context recall is not yet computed.

6. No LangGraph integration yet
- Orchestration is still Python control flow.
- workflow.py remains a placeholder.

7. API key visible in terminal history
- .env has a real OpenRouter key (sk-or-v1-...).
- This key should be rotated.

12. RECOMMENDED NEXT STEPS FROM THIS EXACT STATE
======================================================================

These are the best next engineering steps in order.

1. Singleton pipeline in FastAPI
- Move TreeQAPipeline construction to module startup:
    _pipeline: TreeQAPipeline | None = None
    @app.on_event("startup")
    def _startup(): global _pipeline; _pipeline = TreeQAPipeline()
- Then reuse _pipeline per request instead of rebuilding.
- This eliminates per-request disk I/O for loading index files.
- Effort: Tiny (1-2h)

2. Add semantic retrieval
- Integrate sentence-transformers or a local embedding model for vector search.
- Replace lexical scoring in LocalVectorBackend with cosine similarity on embeddings.
- Saves retrieved chunks to a FAISS or SQLite+vec index.
- Effort: Medium (3-5 days)

3. Run the benchmark with sample.jsonl
- Run: python -m treeqa.eval --dataset data/benchmark/sample.jsonl
- Establish a EM/F1 baseline on local questions.
- Effort: Tiny (already runnable — just needs a live key)

4. Integrate HotpotQA devset
- Download the distractor dev set (JSON) and convert to JSONL for the eval runner.
- Run python -m treeqa.eval --dataset data/benchmark/hotpotqa_dev.jsonl --limit 100
- Gives a meaningful multi-hop retrieval baseline.
- Effort: Small (1 day)

5. Replace recursive pipeline with a LangGraph workflow
- Define nodes: decompose -> retrieve -> generate -> validate -> correct (conditional loop).
- Replace src/treeqa/workflow.py placeholder with a real CompiledGraph.
- This is the largest architectural step toward the spec's intended design.
- Effort: Large (>1 week)

6. Add NLI-based validation
- Use a cross-encoder (e.g., cross-encoder/nli-deberta-v3-base) as a faithfulness signal.
- Add as a second validation path alongside the LLM judge.
- Effort: Medium (3-5 days)
